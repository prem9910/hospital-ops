01 · OverviewProject & Tech Stack

Project Overview

A cloud-synced operations command center designed for hospitals that need to coordinate daily tasks, manage staff across departments, and never lose a single handover.

🎯 The Problem It Solves

Hospital operations involve dozens of recurring tasks, inter-department dependencies, and shift-based handovers that historically live in WhatsApp groups, registers, and memory. Hospital Ops replaces all of these with a single, real-time, role-aware platform that anyone on staff can use from any device.

💎 Key Capabilities

⚡ Real-time Sync

Postgres realtime subscriptions — a change made on one device appears on every other device within seconds.

📴 Offline-First

LocalStorage cache keeps the app fully functional without network, syncs back to Supabase when reconnected.

🔁 Recurring Tasks

7 frequency types (Daily, 15-day, Monthly, Quarterly, Half-yearly, Yearly, Delegation) with parent/child template pattern.

🔔 Bell Notifications

Main admin sees all system activity (task done, handover, delegation, dept change) in a unified notification bell.

🔐 Role-Based Access

3 roles (Main Admin / Admin / Staff) with 22 fine-grained permissions toggled per admin user.

📤 Excel Exports

Every list page has date-range Excel export using the SheetJS XLSX library — ready for compliance audits.

02 · RolesUser Roles & Permissions

User Roles

Three distinct roles. Each one has its own login flow, navigation, and capability set. Permissions for Admin are toggleable per user.

👑 Main Admin SUPERUSER
The platform's superuser. Has unrestricted access to every module. Sees a unified bell with all system activity. Cannot be deleted or demoted by anyone except themselves.
  • Full CRUD on every entity (tasks, employees, departments, handovers, delegations, issues)
  • Bell Notifications — sees ALL notifications (task completed, handover accepted/rejected, dept change requests, delegation status changes, issues reported)
  • Approves Extensions on delegations and overdue tasks
  • View Full Activity Log across all departments
  • Manage Other Admins — create, suspend, and assign permissions
🛡️ Admin CONFIGURABLE
Delegated administrators. Can do everything within their granted permissions. Permissions are toggled individually by the Main Admin from the Admins page.
  • 22 Individual Permissions covering Tasks, Issues, Employees, Departments, Handover, Delegation, Checklists, MIS, Trash
  • Scope-based visibility — admin sees only data they manage, unless granted all_task_details
  • Bell notifications scoped to events relevant to their department
  • No access to other admins' permission settings or role management
👤 Staff EMPLOYEE
Frontline users — nurses, technicians, housekeeping, etc. Log in with their name + department password. See only their own tasks, handovers, and delegations.
  • My Tasks — pending and overdue tasks assigned to them
  • Self-Assign Task — staff can pick a task from a checklist and schedule it themselves
  • My Handover Inbox — accept or reject incoming handovers with remarks
  • My Delegations — accept, mark done, or request extension
  • Report Issue — file a problem with priority and department
  • All Issues (read-only) — see what's been reported by others

05 · TasksManage Tasks & Recurring Engine

📋 Manage Tasks

The most complex module. Admin creates a task template, picks a frequency, and the engine auto-generates child tasks. Staff complete them; new ones appear automatically.

🖼️
SCREENSHOT — Manage Tasks
Task list with frequency pills · Filters · Actions
Manage Tasks grid showing all tasks with frequency badge, assigned employee, due date, and quick actions.

🔁 Recurring Task Engine

Tasks are created as parent templates with a parentTaskId linking to generated child instances. When a child is marked done, the engine automatically generates the next child based on the parent's frequency:

FrequencyLabelAuto-Cycle Behavior
dailyDAILYNext instance = schedDate + 1 day
15-dayHAR 15 DINNext instance = schedDate + 15 days
monthlyMONTHLYNext instance = schedDate + 1 month
quarterlyQUARTERLYNext instance = schedDate + 3 months
half-yearlyHALF-YEARLYNext instance = schedDate + 6 months
yearlyYEARLYNext instance = schedDate + 1 year
delegation📤 DELEGATIONNo auto-cycle — single-shot task

🔄 Extension Workflow

  • Staff Request — if a task can't be completed on time, staff request an extension with a reason and a new proposed date
  • Admin Approval — extension shows up in admin's "Extension Pending" panel
  • Auto-Reschedule — when approved, the task's schedDate is updated to the new date; future recurring cycles use the new date
  • Activity Logged — every extension event is appended to the task's activityLog array
Smart Overdue Backstop: A task whose schedDate is in the past is automatically shown in "My Tasks" pending list with a red ⚠️ OVERDUE badge — even if its frequency hasn't yet triggered a new child instance. This ensures nothing falls through the cracks.

The "What Shows as Pending" Logic

A task appears in Pending when all of these are true:

  • Assigned to me — case-insensitive match against assignedTo array
  • Status is pending — not started, not done
  • Not a grandchild — i.e. not an auto-generated sub-task
  • Today ≥ schedDate — the scheduled date has arrived (or is in the past)

Visual Indicators

SituationDisplay
Scheduled today📅 TODAY
Past schedDate, not done⚠️ OVERDUE — Due: <date>
Future schedDateHidden from pending (shown in Upcoming)
DoneMoves to Completed tab, greyed out
Counts in App Bar: The header's "My Tasks" badge shows the count of tasks in the Pending tab — including overdue. This way staff see their workload at a glance without opening the page.

20 · SetupGetting Started

🚀 Getting Started

From a fresh clone to a running local instance in under 5 minutes.

Prerequisites

  • Node.js 18+ and npm 9+
  • A Supabase project (free tier is enough)
  • A modern browser (Chrome 110+, Firefox 110+, Safari 16+)

Install & Run

# 1. Clone the repository
git clone https://github.com/prem-prakash/hospital-ops.git
cd hospital-ops

# 2. Install dependencies
npm install

# 3. Set up environment variables
cp .env.example .env.local
# Edit .env.local with your Supabase URL and anon key

# 4. Apply database schema
# Open Supabase SQL editor, paste SQL_SCHEMA.sql, run it

# 5. Start the dev server
npm run dev

# 6. Open the app
# http://localhost:5173
# Default Main Admin login is in src/constants/index.js

Production Build

npm run build # builds to /dist
npm run preview # serves the build locally
# Deploy /dist to any static host (Vercel, Netlify, Cloudflare Pages)

First Login

  1. Navigate to http://localhost:5173
  2. Log in as Main Admin using the credentials in src/constants/index.js
  3. Go to Departments and create your hospital's departments
  4. Go to Staff and add employees to each department
  5. Optionally create Admin users with scoped permissions
  6. Start assigning tasks!

Schema Migration Pattern

New columns are added via idempotent ALTER TABLE ... ADD COLUMN IF NOT EXISTS migrations inside SQL_SCHEMA.sql so re-running the file is safe:

ALTER TABLE tasks
  ADD COLUMN IF NOT EXISTS extensions JSONB DEFAULT '[]';

ALTER TABLE tasks
  ADD COLUMN IF NOT EXISTS parent_task_id TEXT;

Realtime Publication

The Supabase realtime channel is enabled per-table. When hops-tasks changes, every connected client receives a postgres_changes event within ~200ms.

23 · Real-timeSync & Offline Cache

⚡ Real-time Sync

The most technically interesting part of the system. Hospital Ops combines three storage layers to deliver a fast, offline-capable, multi-device experience.

Three-Layer Architecture

1️⃣ Supabase (Source of Truth)

PostgreSQL tables with realtime publication. Every write hits Supabase first via the REST API.

2️⃣ LocalStorage (Offline Cache)

Every successful fetch is mirrored to hops-xxx keys. App reads from cache first for instant render.

3️⃣ In-Memory (Live State)

React Context holds the live array. Realtime subscriptions patch the array in place.

Sync Race Condition Fix

A subtle bug existed: when a user creates a task locally, the optimistic state hits the UI immediately. But the realtime channel then fires an event that overwrites the local state with stale data. The fix uses a hops-last-local-write timestamp:

// On save:
ls.set('hops-last-local-write', { key: hopKey, at: Date.now() });

// On realtime event:
if (eventTime < lastLocalWriteAt) {
  // Preserve local pending rows — don't let server race overwrite them
}
Offline Mode: If the network drops, the app keeps working from the LocalStorage cache. When network returns, queued writes flush to Supabase and the cache re-syncs.

24 · FutureSecurity & Roadmap

🛡️ Security & Roadmap

Current security posture and the planned evolution of the platform.

Current Security

🔑 Anon-Key Supabase Auth

Only the Supabase anon key is shipped to the browser. All write paths go through Row-Level Security policies.

🛡️ Role-Based UI Guards

Every page checks currentRole and hasPerm(...) before rendering sensitive actions.

⏱️ Inactivity Auto-Logout

5-minute inactivity triggers logout to prevent unattended access.

🚫 No Service-Role in Client

The Supabase service-role key must NEVER be placed in public/ or bundled into the client. Diagnostic scripts must read from server-side env vars.
